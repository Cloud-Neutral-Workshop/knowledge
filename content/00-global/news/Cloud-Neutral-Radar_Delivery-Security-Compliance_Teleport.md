# Cloud-Neutral 资讯雷达｜交付与安全合规 · Teleport

## Teleport

Teleport 是 Cloud-Neutral Tool Map 中 **交付与安全合规（Access & Audit Control）** 这一层里，极具代表性的现代访问控制平台。  
它并不是传统意义上的“堡垒机”，而是从工程视角重新回答了一个老问题：

> **在现代基础设施中，谁在什么时间，以什么身份，访问了什么。**

在 SSH、Kubernetes、数据库、Web 控制面高度碎片化的现实下，Teleport 选择用**统一身份、短期凭证和强审计**来解决访问与合规问题。

---

## 项目主要特性

- 基于身份的统一访问控制（SSH / K8s / DB / Web）
- 短期证书替代长期密钥
- 内建 RBAC 与审计日志
- 支持 SSO / OIDC / MFA
- 云中立，可运行于任意环境

Teleport 的核心设计不是“方便登录”，而是：  
**把访问行为变成一种可控、可审计的工程资产。**

---

## 优缺点

| 优点 | 局限 |
|---|---|
| 统一多协议访问模型 | 部署与理解成本较高 |
| 无长期密钥，安全性强 | 不适合极简环境 |
| 审计能力原生 | 学习曲线陡于传统堡垒机 |
| 云厂商中立 | 小规模场景略显“重” |

---

## 适用场景

| 适合 | 不适合 |
|---|---|
| 多集群 / 多环境访问控制 | 单机 / 极小规模系统 |
| 合规 / 审计要求高 | 临时、低风险环境 |
| Dev / Ops / SRE 统一入口 | 只需要简单 SSH 跳板 |
| 云原生与混合环境 | |

---

## 工程判断

Teleport 并不是“更好的 SSH 工具”，而是**访问控制工程化的结果**。  
它清晰地站在这样一个工程边界上：

> **访问不是权限配置问题，而是身份、时间与行为的组合问题。**

在现代交付体系中，一个清晰的位置是：

> Terraform / Pulumi 定义资源边界，  
> **Teleport 控制谁可以进入这些边界**，  
> 审计系统负责长期留痕与追责。
