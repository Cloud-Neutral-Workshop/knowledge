# Cloud-Neutral 资讯雷达｜交付与测试 · DevTools MCP

## DevTools MCP

DevTools MCP 通常指 **基于 Chrome DevTools Protocol 的 MCP（Model Context Protocol）Server 实现**。  
它并不是传统意义上的测试工具，而是 Cloud-Neutral Tool Map 中**测试与自动化智能化方向**的重要代表。

它关注的问题并不是“是否通过”，而是：

> **系统中是否还存在尚未被理解和覆盖的行为路径。**

---

## 项目主要特性

- 将浏览器 DevTools 能力暴露给 LLM / Agent
- 可观察 DOM / Network / Console
- 动态决策而非预定义脚本
- 面向探索、分析与生成
- 强实验性与可扩展性

DevTools MCP 的核心价值不是稳定执行，而是：  
**探索与理解。**

---

## 优缺点

| 优点 | 局限 |
|---|---|
| 强探索与理解能力 | 可复现性较弱 |
| 适合复杂系统 | 工程成熟度不一 |
| 可生成测试思路 | 不适合作为质量门禁 |
| AI 友好 | 结果不确定 |

---

## 适用场景

| 适合 | 不适合 |
|---|---|
| 智能测试生成 | CI 硬性校验 |
| 复杂系统探索 | 合规 / 审计场景 |
| QA 辅助 | 性能基准 |
| Agent 驱动自动化 | |

---

## 工程判断

DevTools MCP **不是 Playwright 的替代品**，也不是 k6 的竞争者。  
它清晰地站在一个新的工程责任边界上：

> **为测试体系提供“发现问题的能力”，而不是“给出结论的能力”。**

在现代交付体系中，一个清晰的分工是：

> **DevTools MCP 负责发现可能性，  
> Playwright 负责固定事实，  
> k6 负责验证极限。**

---

> 如果说传统测试关心“对不对”，  
> DevTools MCP 关心的则是：  
> **“我们是不是还没问对问题。”**
